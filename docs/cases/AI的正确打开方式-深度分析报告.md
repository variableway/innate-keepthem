## 一、内容概述

该视频提出一个核心论断：**大多数 AI 学习者被困在"名词层"——追逐工具/概念的名称（RAG、Agent、LangChain 等），却忽视了真正决定 AI 能力的"动词层"——完成任务所需的操作能力（上下文管理、任务拆解、诊断、评估等）。**

视频以 RAG（检索增强生成）为例详细展开，展示了一个典型的名词学习者 vs 动词学习者的差距。最终推销 Superlinear Academy 的课程——据称是唯一系统教授这些"动词"的地方。

## 二、核心框架：名词 vs 动词

### 2.1 名词清单（学习者追逐的概念）

| 名词 | 含义 | 典型学习方式 | 问题 |
|------|------|-------------|------|
| **RAG** | 检索增强生成 | 学 pipeline：切分→embedding→向量库→top-k→prompt | 真实场景完全不是 pipeline 能解决的 |
| **Fine-tune** | 微调 | 用小数据集调整模型 | 以为有数据就能解决一切 |
| **Agent** | 智能体 | 学模型调用工具 | 不懂何时该用、何时不该用 |
| **LangChain** | LLM 框架 | 学 API 和链式调用 | 框架本身是 counterproductive |
| **Harness Engineering** | 提示工程平台 | 学工具用法 | 关注工具而非结果 |
| **MCP** | Model Context Protocol | 学协议规范 | 追逐新标准而忽视实际需求 |

### 2.2 动词清单（真正决定能力的东西）

视频提出但**未详细展开**的六个 RAG 动词：

| 动词 | 视频中的简述 | 隐含的深度问题 |
|------|-------------|---------------|
| **上下文管理** | 什么时候给 AI 更多/更少上下文 | 如何量化"足够"？上下文窗口策略？动态注入机制？ |
| **切分 (Chunking)** | 不只是固定 size 切分 | 语义切分、层级切分、任务感知切分？ |
| **判断 (Judgment)** | 哪些环节交给 AI，哪些必须人判断 | 判断标准？置信度阈值？人机协作协议？ |
| **评估 (Evaluation)** | 针对任务本身的 evaluation | 自动化评估框架？ground truth 建立？ |
| **诊断 (Diagnosis)** | 出错时定位 root cause | 诊断方法论？trace 工具？分层排查策略？ |
| **编排 (Orchestration)** | 怎么组织整个任务流程 | 工作流引擎？动态路由？fallback 策略？ |

视频末尾还泛泛提到了：**管理、拆解、判断、路由、诊断、沉淀**——作为通用的 AI 工作手艺。

## 三、视频的核心"卖关子"——没说清楚的部分

视频巧妙地完成了一个叙事闭环：**描述问题 → 告诉你问题很大 → 告诉你我知道答案 → 答案在我的课程里**。以下是被刻意模糊化但视频暗示"非常重要"的内容：

### 3.1 具体操作被抽象化

视频说"我会用非常详细的例子去跟大家讲清楚"，但实际上视频中**没有给出任何一个完整的动词操作示例**。RAG 的六个动词只提了名字，连一个具体场景都没有展开。

### 3.2 "手艺"隐喻的价值和局限

视频用高尔夫、拳击做类比——"高手也是一个刺拳，你也是一个刺拳，但你看不到差距"。这个隐喻很好，但：
- 高尔夫/拳击有明确的教练反馈机制（球飞多远、打中没）
- AI 工作的"结果好坏"缺少客观评价标准
- **视频没有说明如何在 AI 场景下获得类似教练的反馈**

### 3.3 课程的"不可 pip install"特性

视频强调这些动词"不能 pip install、不能写进简历技能栏"——这是精准的市场定位。但同时：
- 如果不能在简历上证明，面试中如何展示？
- 视频暗示"我们的课程能解决"，但未说明机制

### 3.4 RAG 六动词的具体挑战

以"上下文管理"为例，视频只说了"什么时候该给更多上下文，什么时候应该减少上下文"。但实际上需要回答：
- 如何检测上下文是否"够"？（token 利用率？答案准确率？）
- 动态上下文的检索策略是什么？（keyword → semantic → hybrid？）
- 长对话中的上下文窗口滑动策略？
- 多模态上下文（代码+文档+对话历史）如何组织？

## 六、VTT → 深度研究报告的 Skill 可行性

### 6.1 核心能力需求

从本次分析过程提炼，"VTT 深度分析 Skill" 需要：

| 步骤 | 能力 | 状态 |
|------|------|------|
| 1. VTT 解析 + 清洗 | `vtt.Parse()` + `PlainText()` | ✅ 已实现 |
| 2. 文本分段 | `SegmentByDuration()` | ✅ 已实现 |
| 3. 内容总结 | LLM 摘要 | 🔜 Phase 3 |
| 4. 要点提取 + 时间戳 | LLM keypoint extraction | 🔜 Phase 3 |
| 5. 概念识别 | 提取专业术语/人名/产品名 | 🆕 需新增 |
| 6. 网络调研 | 搜索补充信息（人物、组织、概念） | 🆕 需新增 |
| 7. 差距分析 | 识别"说了什么"vs"没说什么" | 🆕 需 LLM |
| 8. Markdown 报告生成 | 模板化输出 | 🆕 需模板 |

### 6.2 实现路径

```
方案 A: CLI 扩展（Go）
  vYtDL analyze --mode research video.vtt
  → vtt 解析 → LLM 摘要 → 概念提取 → web_search → 合成报告

方案 B: Python Skill（DeepSeek Skill）
  独立的 Python 脚本，集成 vtt parser + LLM API + 搜索
  → 更灵活，可复用已有 Python 生态

方案 C: 混合
  Go CLI 做解析 + 清洗 → 输出结构化 JSON
  Skill 消费 JSON → LLM 分析 → web_search → Markdown 报告
```

### 6.3 推荐方案

**方案 B**（Python Skill）最务实：
- 可以利用 DeepSeek 的 `web_search` / `fetch_url` 能力
- Python 的 LLM 客户端生态成熟
- Skill 可以定义标准化的分析模板（prompt chain）
- 本次分析的 70% 工作（内容总结、概念提取、差距识别）本质上是 LLM prompt engineering

### 6.4 Skill 设计概要

```yaml
skill:
  name: vtt-deep-research
  description: 从 VTT 字幕生成深度研究报告
  inputs: vtt_file_path
  outputs: markdown_report.md
  
  pipeline:
    - vtt_parse: 解析 + 清洗 → 全文 + 分段
    - concept_extract: LLM 提取 名词/动词/人名/组织
    - web_research: 对提取的实体进行搜索补充
    - gap_analysis: LLM 识别"说了什么 vs 没说什么"
    - report_gen: 按模板生成 Markdown 报告
```

## 七、结论

### 7.1 视频评价

**信息密度**：高。22 分钟覆盖了一个完整的认知框架。

**实操价值**：中等偏低。视频完成了"问题意识唤醒"但几乎没有可操作的具体指导。

**商业目的**：明确。视频是 Superlinear Academy 课程的高级内容营销。

### 7.2 最值得深入研究的方向

1. **RAG 六动词的具体实现**——这是视频最接近给出干货的部分，但仍然只是"提了名字"
2. **AI 工作的 evaluation 方法论**——视频强调这是关键但没展开
3. **诊断 AI 错误的系统方法**——视频说高手能 debug，但没说怎么 debug
4. **Superlinear Academy 课程的实际内容**——如果课程真的给出了这些动词的详细拆解，值得进一步调研

### 7.3 关于 Skill 的建议

**可以立即启动**。Phase 1+2（VTT 解析+清洗）已完成，Phase 3（LLM 集成）可以作为 Skill 的第一步。核心工作是设计好 prompt chain——概念提取 → 差距识别 → 调研 → 报告合成的模板。本次分析本身就可以作为 Skill 的第一份 sample output。

## 附录：清洗后的完整文本

> 注：原始 VTT 文件末尾有 YouTube 自动生成的拼接错误和重复段落，已手动清理。

最近我一直在想一个问题：为什么很多人学 AI 越学越焦虑，而且他们的水平其实被限制在了一个地方，很难提高。

看上去这个人懂很多东西，但是实际他在工作中使用 AI 的时候，非常的不靠谱，很多东西他是做不好的。但是你问他的话，什么新概念他都知道，他都能给你讲出来道理——fine-tune、RAG、各种 Framework、LangChain、包括 Harness Engineering，他全都能给你讲。

而且这些人，他可能看了很多个 AI 工作流，看到了很多大佬是怎么样子 prompt 的。他在做一些任务的时候呢，他偶尔也能把 AI 用得很好——写个文案、写个代码，都能感觉到 AI 好像确实是很厉害的。

但是进入到真实工作的时候，他们就抓瞎了。因为在真实工作中，你的工作不是这样的一个一个片段，你是一个非常连续的、要不断推进的东西。你的代码仓库里边可能有很多历史的包袱，你的上下文可能是不全的，模型针对不同的新出来的任务都需要针对这些任务本身去设计答案，而不是复制一个现成的别人的方法就可以把问题解决的。

你还会发现在这个使用的过程中，别人给你展示的是一个 happy path，但是在你的使用中，它会偷懒、会忘、会答非所问、会给你 hallucinate，会在一个你以为不会犯错的地方突然给你犯个错。

那这个时候很多人的第一反应就是：是不是我的 prompt 没有学好啊？是不是我还有很多新的 fancy 的 technique 我没有学会？我是不是应该换个模型、换个工具？是不是应该去把那些概念再重新再学习一下？

于是呢，他们就去继续学习下一个名词，而离把这个任务真的做完越来越远。

因为这些人 AI 能力卡住，问题恰恰出在他们去追逐名词。而真正决定一个人 AI 用得好不好的，是那些**动词**。

RAG 是名词，Harness Engineering 是名词。它们当然有价值，也值得我们去了解，但其实这些东西真的要去学的话，5 分钟你跟 AI 就都能学会了。

真正厉害的人和普通使用者之间的差距，不是在于多学了一些名词、多知道一些新东西，而是在于他处理任务的方式完全不一样。

比如说：
- 他知道什么时候该给 AI 更多上下文，什么时候应该减少上下文
- 一个复杂任务应该怎么样子拆给 AI
- 哪些环节应该交给 AI，哪些环节必须要有人来判断
- 针对这个任务本身的 evaluation 到底我们应该怎么看
- AI 怎么样子才能保证它去非常好的 follow instruction
- 当整个任务出错了以后，他们能 debug，能 identify root cause 并且把它解决

这些东西听起来就没有那么"技术"了，不像 MCP、LangChain 那样一听就是个技术。他们没有 GitHub 仓库——除了我们的课，基本上网上也没有人去教这些东西。他甚至也不能 pip install 变成一个 package。

而且这些东西你学了以后你真的变强了，你还很难把它写到你的简历的技能栏里边。所以大家不重视。

但是最终决定你 AI 能力的，恰恰就是这些动作，是这些动词。

---

**为什么大家更愿意学名词？**

第一个原因：名词是有边界的，它给你错误的进步的感觉。你学了 RAG 你就以为可以解决这些问题了，学了 agent 你就知道模型可以调用工具了。它让你觉得前后的差距特别明显。

第二个原因：名词可以被证明。我学过 RAG、我做过 RAG——可以写到简历上。checklist 打上了以后，你就不用再继续学下去了。

所以我们脑子里有两个货架：
- 第一个货架：RAG、Agent、Harness Engineering——看起来很硬
- 第二个货架：上下文管理、任务拆解、人机协同、结果确定性——看起来很软

大家更愿意去投入第一个货架。

---

**以 RAG 为例的深度拆解**

很多人学 RAG 的方式是从 pipeline 开始的：文档切分→做 embedding→存进向量数据库→top-k 召回→塞进 prompt→让模型生成答案。听起来非常完整，让人很心安。

学到这，很多人就有一个危险的错觉——自己会做 RAG 了。

可是到了真实的应用场景，这个流程根本没用，而且非常 counterproductive。

真实企业中需要面对的问题：
- 用户问的问题到底应该召回哪些资料？5段还是20段？
- chunk 太大可能把答案切分开，太小可能丢失上下文
- 用户换一种说法问同样的问题，检索命中率下降了怎么办？
- 文档过期了，系统还在返回旧内容怎么办？
- 不同部门有不同的权限怎么办？
- 答案的引用到底准不准？什么才是 ground truth？
- 这次的回答错了——到底是检索错了、生成错了、prompt 错了、还是用户意图理解错了？

用名词的方式学 RAG，你关心的是 pipeline 里边有哪些零件。用动词的方式学 RAG，你关心的是这个系统怎么样子在一个工作环境中稳定地工作。

---

**RAG 概念对应的六个动词能力**

1. **上下文管理**
2. **切分**（不只是 chunking）
3. **判断**
4. **评估**
5. **诊断**
6. **编排**

每一个展开都有非常深的内容——这些基础的 RAG pipeline 有用的东西早就被平台吸收了，企业愿意付出溢价的，正好是这个 pipeline 之上的能力。

---

**回到抽象主线**

RAG 只是一个例子。换成 Agent、Harness Engineering、AI 编程、AI 写作、AI 知识库——都是一样的。

- 初学者关心的是"我该学哪个工具"；高手关心的是"这个任务该如何组织"
- 初学者关心的是"怎么样子写一个更好用的 prompt"；高手关心的是"这个任务需要什么样的上下文结构"
- 初学者关心的是"哪个模型最强"；高手关心的是"哪个模型、哪个工具、哪个流程更适合这个任务"
- 初学者问"为什么 AI 又错了"；高手问"到底哪里出错了"

**高手的动作围绕：管理、拆解、判断、路由、诊断、沉淀。**

最后导致大家用同样的模型、同样的工具，质量的差距特别大。有很多程序员抱怨 AI 不好用——很多时候就是你自己太菜了，停留在非常浅的名词里边，你甚至根本就不知道这些动词的存在。

---

**为什么会发生这种事？**

第一，工具的进步很容易被看见，但动作的进步不容易被看见。

第二，人只能看到自己当下的瓶颈。那些动词的瓶颈你是看不到的。只有你真的把这些东西做完了、克服了这些瓶颈，你才有可能看到这些动作的重要。

这就是为什么鸭哥（课代表立正）这么稀缺——他不光完全走过了这些路，还能把这些东西教出来，具象化、深入浅出。

---

**总结**

为什么你学了很多 AI 越来越焦虑，却一直没有办法把 AI 用好？

因为大家一直看到了上面这条线——各种各样的名词，没有看到下面这个面——真正的动词和动词展开所对应的你要做好的那些事。

真正的入门不是你去追逐最新的概念，而是你开始看到下面的这些动作，开始去提高你下面的这些动作。

工具会变、模型会变、框架会变——这些名词天天都会变。但是这些动作是不怎么变的。课程到现在 12 期，每期都在更新，但是 70% 的动作我们发现是不变的。而且随着模型变强，它们是越来越有复利的。

这些沉淀的经验和认知，都会有长期价值、长期复利——因为这些本质，就是我们 AI 时代的工作手艺。

下一次当你觉得自己学了很多 AI 但是没有入门的话，先问一下自己：我看没看到这个手艺背后对应的那些动词？我去哪可以学习到这样更多的动词？我去哪可以提高我这些动词？

这可能才是真正的入口。
